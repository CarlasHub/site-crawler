Crawl a site and review the results in one report.

Cat Crawler starts from a homepage or scoped path, follows internal routes, and groups the findings into clear sections for redirects, parameter handling, soft failures, URL patterns, and issue impact.

The bookmarklet opens the Cat Crawler app in a floating panel and passes the current page URL into the app. It does not replace the backend crawler.

What it is

A React frontend plus Node backend for crawl and navigation validation, with an optional bookmarklet installer for launching the app from the page you are already on.

What it is good for

  • Internal site QA and regression checks
  • Redirect and routing review after content or platform changes
  • Parameter handling checks for search and filtered results pages
  • Quick launch from the current page with the bookmarklet

Current limits

  • Same-host crawling only
  • Backend hard caps: up to 300 pages and concurrency 6
  • Staging and production require Firestore-backed job state
  • GitHub Pages only hosts docs and the bookmarklet loader, not the crawler backend

What the tool does

It crawls internal navigation from a starting URL, records referrers, respects robots.txt, follows redirects, and groups results into reviewable report sections.

Who it is for

Developers, QA teams, SEO reviewers, and site owners who need a structured crawl report instead of a manual click-through.

What it does not claim

It does not replace human judgement, and the GitHub Pages site does not provide a hosted crawler service by itself.

How it works

Use it in four simple steps

The UI is built around a crawl job, progress polling, and grouped report sections. The bookmarklet just helps you start from the page you already have open.

1

Choose a starting URL

Use a homepage or a scoped path such as /en if you only want one section of a site.

2

Set crawl rules

Add exclude paths, path limits, max pages, concurrency, and optional checks such as parameter audit.

3

Run the crawl

The backend creates a background job, crawls internal routes, and reports progress back to the UI.

4

Review and export

Use the grouped report sections, then export TXT or CSV when you need to share the result.

Feature

Main dashboard and progress

The app shows a single runner view for crawl setup, progress, and the latest crawl state so you can see whether the job is queued, running, complete, or failed.

User value You can start a crawl and understand its current state without switching between separate tools or screens.
  • Shows progress percent, phase, and crawl counts
  • Keeps the latest crawl visible after completion
  • Surfaces failures in the same UI instead of sending you to a separate console

Feature

Crawl runner and scope controls

The main runner lets you seed a crawl, exclude sections, cap noisy paths, and choose whether to enable broken-link checking or parameter audit.

User value You can narrow the crawl to the parts of a site that matter without changing backend code or editing a config file.

  • Path limits are language-agnostic, so /job can also match /en/job
  • The UI clamps max pages and concurrency to the backend caps

Feature

Main validation report

The validation report collects the main issue groups into one place so you can review broken URLs, redirect issues, parameter issues, soft failures, and prioritised impact without switching screens.

User value You get one report entry point instead of having to inspect raw crawl output line by line.
  • Broken URLs are separated from redirect and soft-failure issues
  • Referrers and final URLs are kept visible for triage
  • Impact analysis helps sort repeated or higher-value issues first

Feature

Redirect audit

Redirected routes are expanded into a dedicated audit with step-by-step chain details and flags for loops, multi-hop redirects, dropped parameters, and irrelevant destinations.

User value Redirect problems are isolated from the rest of the crawl so they are easier to diagnose and hand off.
  • Shows redirect status codes and final destination
  • Flags loops and long redirect chains
  • Shows when a redirect drops query parameters

Feature

Parameter audit

Parameter audit sends a small set of querystring variations to the site and records whether parameters are preserved, dropped, or redirected in unexpected ways.

User value It helps catch fragile routing or filtering behaviour that a normal crawl would not reveal.
  • Useful for search pages, filter pages, and paginated listings
  • Separates HTTP errors from parameter-preservation problems
  • Shows base URL, tested variant, and final URL together

Feature

Soft failures, URL patterns, and duplicate candidates

Cat Crawler also highlights pages that look broken despite returning success, structural URL anomalies, and possible duplicate content groups.

User value You can review issues that are easy to miss when you only look at status codes.
  • Soft failures can include missing content, error text, or failed API calls
  • Pattern audit helps spot inconsistent naming or legacy/current path pairs
  • Duplicate candidates group query variants and language variants

Bookmarklet

Launch Cat Crawler from the page you are already viewing

The bookmarklet loads the current docs/bookmarklet.js from GitHub Pages, reads the configured app origin, and opens the deployed app in an iframe with the current page URL prefilled.

What it does

  • Injects a floating button and panel into the current page
  • Loads the Cat Crawler app from configured app origin
  • Passes the current page URL into the app as the crawl seed
  • Lets you drag the panel and resize it from four corners

What it depends on

  • The public docs config must contain the correct appOrigin
  • The target app must actually be deployed at that origin
  • The bookmarklet is an app launcher, not a self-contained crawler runtime
  • For a new environment, regenerate docs/config.js before publishing

Technical notes

Current run model, deployment model, and limits

This section keeps the operational facts close to the public docs without trying to replace the README.

Architecture

  • React frontend served as a built Vite app
  • Express backend for crawl jobs and result APIs
  • GitHub Pages hosts docs and the bookmarklet loader
  • Bookmarklet mode opens the main app in an iframe

Current backend constraints

  • Background-job API is the normal UI path
  • Default and maximum pages: 300
  • Default and maximum concurrency: 6
  • Per-IP rate limits are enforced by the backend

Production deployment notes

  • Local development defaults to file-backed job state
  • Staging and production must use JOB_STATE_BACKEND=firestore
  • All production instances must share the same Firestore backend
  • Docker is the supported production build path in CI

What to treat carefully

  • Soft-failure and pattern sections are review aids, not absolute truth
  • GitHub Pages alone does not host the crawler app
  • The bookmarklet must point at a real deployed app origin
  • Same-host crawling is intentional and enforced
Setup and deployment

Start locally, then configure the docs for the environment you are publishing

Local setup

  • Run npm ci in frontend/ and backend/
  • Build the frontend with npm run build in frontend/
  • Start the backend with npm start in backend/
  • Local bookmarklet config defaults to http://localhost:8080

Docs publish path

  • Use APP_ENV and BOOKMARKLET_APP_ORIGIN with scripts/write-public-config.mjs
  • Validate the docs installer with scripts/validate-public-docs.mjs
  • Keep docs/index.html and docs/landing.html aligned
  • Read the README for full deployment and Docker details
